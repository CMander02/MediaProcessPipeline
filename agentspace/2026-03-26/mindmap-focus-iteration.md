# 树状图/导图聚焦交互修复记录

日期：2026-03-26

## 背景

结果页右侧导图希望支持“点击节点后自动聚焦”，但目标并不是简单的 `fit()` 或把整棵树重新渲染成独立子树，而是：

1. 点击节点后，展开逻辑保持原样。
2. 相机只按局部语义重新取景。
3. 取景完成后，其他节点仍然保留在画面里。
4. 视觉效果上像“先按局部子树摆镜头，再把其余节点拼接回来”。

用户反复强调的典型场景：

- 点击 `Part 1: 嘉宾背景与早期经历` 时，希望优先看 `Part 1` 和其一级子节点。
- 点击 `嘉宾背景：翁家翌` 时，希望优先看：
  - `嘉宾背景：翁家翌`
  - `2022年加入OpenAI，是系列模型的核心贡献者`
  - `贡献领域：强化学习、后训练、基础设施`
  - `参与模型：ChatGPT、GPT-4o、GPT-5等`
  - `加入OpenAI前的开源贡献`

而不是被父层、兄弟层或更远处的长文本节点把取景拉散。

## 已尝试方案

### 1. 点击节点后直接重绘为局部子树

做法：

- 点击任意节点后，把当前节点 clone 出来，整张图重绘成以该节点为根的新 markmap。

问题：

- 交互上会“切图”，用户继续点时语义断裂。
- 原图上下文丢失太多。
- 与用户期望的“整棵树还在，只是视角变了”不一致。

结论：

- 放弃。

### 2. 保留整棵树，只做节点展开 + 平移/缩放

做法：

- 继续用 markmap 原始树。
- 点击节点后调用 `toggleNode`。
- 然后尝试用 `centerNode` / `rescale` 做相机移动。

问题：

- `centerNode` 对的是节点文本框中心，不是节点的连接点/展开点。
- 长文本节点尤其明显，看上去“中心没对准”。

结论：

- 只能作为过渡方案，不足以满足需求。

### 3. 用“父链 + 祖先兄弟 + 当前节点 + 一级子节点”决定取景

做法：

- 参与取景的节点扩大到父链以及祖先层的一级兄弟。

问题：

- 祖先层兄弟节点，尤其根层大分支，会显著拉大包围盒。
- 结果是缩放过于保守，看起来几乎没移动。
- 与“局部优先”的主观感受相反。

结论：

- 方向不对。

### 4. 用“父节点 + 当前节点 + 一级子节点”决定取景

做法：

- 收窄到更局部的骨架，仅保留父节点、当前节点、当前节点一级子节点。

改进：

- 比祖先层方案明显更聚焦。

问题：

- 对于右侧长文本节点，父节点仍会干扰取景宽度。
- 点击的是二级/三级节点时，父节点文本可能仍然过长。

结论：

- 接近目标，但还不够“局部根视图”。

### 5. 用“当前节点 + 一级子节点”决定取景

做法：

- 完全按当前点击节点当作局部根来决定相机。
- 其他节点不参与缩放计算，但仍保留在最终画面里。

这是目前最接近用户需求的一版，因为它符合：

- 先只考虑局部根和其子一级；
- 先把这些节点安排进屏幕空间；
- 再把其他节点当作额外挂回整棵图上。

问题：

- 目前实现仍然是“用真实全图的现有布局结果，抽取一部分节点来反推相机”。
- 这不等于“先独立布局一个局部树，再把其他节点拼回去”。
- 所以当右侧某些文本过长时，虽然已经改善，但仍然会出现视觉不够理想的情况。

## 当前结论

当前实现只是“局部节点驱动的相机近似”，不是用户真正要的那种：

> 先把局部根及其一级子节点单独按屏幕空间布局，再把其他节点拼回去。

也就是说，问题暂时还没有真正解决。

## 根因

现有 `markmap` 布局是一次性对整棵树算出来的。

我们目前做的所有修补，本质上都只是：

1. 在整棵树已经布局完成后，
2. 从中挑出一部分节点，
3. 用这些节点估算一个缩放和平移。

但用户真正想要的是：

1. 先构造一个“局部视图树”；
2. 只对这个局部视图树做布局；
3. 让这个局部布局占据主要屏幕空间；
4. 再把其余节点按弱化/附着的方式放回画面。

这两件事在视觉效果上差很多。

## 推荐的下一步实现

最靠谱的方向不是继续微调包围盒，而是把“取景”升级为“局部布局”。

建议拆成两层：

### A. 定义聚焦子树

点击节点 `N` 后，构造一个临时聚焦树：

- 根：`N`
- 子：`N.children`

可选保留一个弱化的上游锚点：

- `N.parent` 只作为一条左侧导入线或一个弱化节点存在
- 但不要让它参与主布局宽度计算

### B. 单独布局聚焦树

对这个临时聚焦树独立计算：

- 主节点位置
- 一级子节点位置
- 合适的横向文本空间

然后再把完整树的其他节点以“弱上下文”的方式绘制出来，例如：

- 透明度降低
- 不参与主缩放
- 超出主视野也允许

## 如果继续在现有 markmap 上做最低成本修补

如果暂时不想重写局部布局，最低成本还能尝试的只有：

1. 点击节点后，把连接点落在视口 `40% ~ 45%` 的位置，而不是正中。
2. 对长文本子节点额外增加右侧留白。
3. 当当前节点有一级子节点时，缩放只由 `当前节点 + 一级子节点` 决定。
4. 父节点只参与“视觉连接”，不参与宽度估算。

但这依然只是 patch，不是最终解。
